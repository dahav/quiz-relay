TOGAF-specific rules:
- The TOGAF Standard is a framework to be adapted, not a rigid mandate.
- Distinguish carefully between commonly confused pairs:
  - Request for Architecture Work (triggers cycle, from sponsor) vs Statement of Architecture Work (defines scope/approach, agreed before work begins) vs Architecture Contract (joint agreement on deliverables/quality/fitness-for-purpose).
  - Architecture Definition Document (container for core architectural artifacts) vs Architecture Requirements Specification (quantitative requirements) vs Architecture Roadmap (progression Baseline → Target).
  - Architecture Vision (high-level aspirational view, Phase A) vs Target Architecture (detailed future state).
  - ABB (Architecture Building Block, defines required capability) vs SBB (Solution Building Block, fulfils capability).
  - Enterprise Continuum (classification system) vs Architecture Repository (storage) vs Architecture Landscape (architectures at Strategic/Segment/Capability levels).
  - Initial risk (before mitigation) vs Residual risk (after mitigation).
- Map ADM phase objectives precisely: Phase B = Business gaps, Phase C = Information Systems (Data + Application) gaps, Phase D = Technology gaps, Phase E = Opportunities & Solutions (Transition Architectures, SBBs, value/cost of work packages), Phase F = Migration Planning (approved set of projects with dates), Phase G = Implementation Governance (conformance), Phase H = Architecture Change Management.
- The ADM diagram is a "stylized representation showing essential information flows", not a linear waterfall and not strictly an iteration cycle.
- Practical guidance on applying the TOGAF Standard lives in the TOGAF Series Guides; the Fundamental Content defines the core concepts.
